Introduction

  • Used to speed up retrieval of records in response to certain search conditions
  • Ex: Get all employees details where age = 50 or eid = 329;
  • In real world databases, indexes maybe too large to be handled efficiently
  • Hence some sophisticated techniques can be used
  • Criteria for evaluating the hashing or indexing techniques
    • Access Time
    • Insertion Time - When you insert a record, how much time it requires to modify a structure
    • Deletion Time -
    • Space Overhead - Space required to store the indexes and their records
  • You use either hashing or indexing, not both of them
  • Some times more than one indexing maybe required for a file
  • The attribute/field used for constructing index structure for a file called an indexing field/attribute
  • If the index field is a key, it is called as search key or indexing key

Dense Index

  • In this an index record appears for every data file record

Sparse Index

  • Index records are created only for some data file records.
  • This occupies less space

NOTE: Primary index and clustering index are non dense â†’ sparse

NOTE: Secondary index on non key can be non dense â†’ sparse because of the fact that we can have 1,1 3, 3 and stuff like that

Data Record

  • Similar kind of records are stored in a single file containing blocks
  • These are called data records and will have fields specified on the relation

Index Record

  • Like data records, index records are also stored in database
  • Any index record normally has two fields value and pointer(location address of the record containing key)

How they are stored

Spanned and Unspanned

  • Un Spanned means only complete records can be stored in this block and any record cannot be split into parts
  • Spanned has records split into parts which will thereby save space but might take a lot of time for access because we will need to access more blocks
  • We have 512 byte is block size and each record size is 100bytes, therefore BFR = floor(512/100) = 5
  • This gives number of records per block
  • Now number of data blocks = ceil(502/5) = 101 assuming number of records is 5
  • This has nothing to do with indexing but we have chosen to store the records this way
  • If we need to find a record given an id, we need to access log 2 N blocks to get the data
  • We use Block Access Time because this gives you the time required for disk to ram time

  • Usually BFR_i should be higher as that of BFR since block size is fixed by Operating System but length of each index record is less than length of each data record
  • To search for a record of key 50, we need to access log x + 1 â†’ That extra one comes because of data block

Example 1

  • Assume that we have an ordered file with 80000 records stored on disk. Block size is 512 bytes. Record length is fixed and is 70 bytes
  • PK length is 6 bytes and block pointer is 4 bytes
  • Assume un spanned record organization

Answer

  • No of Records = 80000, Block Size = 512b, Key = 6b, Pointer = 4b
  • Record Length = 70b, Index Record Length = 10b
  • BFR = floor(512/70) = 7.31 = 7 we take floor cause otherwise weâ€™ll cross block size
  • BFR_i = floor(512/10) = 51
  • No of data blocks needed = ceil(80000/7) = 11429
  • No of Index records = No of data blocks = 11429
  • No of index blocks = ceil(11429/51) = 225
  • If no indexing is there data access = ceil(log2(11429)) = 14
  • With indexing weâ€™ll have ceil(log2(225)) + 1 = 8 + 1 = 9

Example 2

  • Assume that we have an ordered file with 130000 records stored on disk. Block size is 1024 bytes. Record Length is fixed and it is 250 bytes. Key Field PK length is 10 bytes and block pointer is 8 bytes. Assume un spanned record organization
  • Design Primary Index on Primary key

Answer

  • BFR = floor(1024/250) = 4
  • BFR_i = floor(1024/(10 + 8)) = 56
  • No of blocks needed = ceil(130000/4) = 32500
  • No of Index records = No of blocks = 32500
  • No of index blocks = ceil(32500/56) = 581
  • No of block accesses = ceil(log2 581) + 1 = 10 + 1 = 11
  • Without indexing = ceil(log2(32500))

Example 3 - Multi Leveled

  • Assume that we have an ordered file with 130000 records stored on disk. Block size is 1024 bytes. Record Length is fixed and it is 250 bytes. Key Field PK length is 10 bytes and block pointer is 8 bytes. Assume unspanned record organization
  • Design Multi level Index on Primary key

Answer

  • Size of disk block = 1024b, Record Length = 250b, Block Pointer = 8b, Key Field = 10b, Total Records = 130000
  • No of records per block (BFR) = floor(1024/250) = 4.096 = 4
  • No of data blocks needed = ceil(130000/4) = 32500
  • Index Record Length = 18b
  • BFR_i = floor(1024/18) = 56
  • No of index blocks in level 1 = ceil(32500/56) = 581
  • No of index blocks in level 2 = ceil(581/56) = 11
  • No of index blocks in level 3= ceil(11/56) = 1
  • Total No of block accesses = no of levels + 1 = 4

B+ Tree Indexing

  • Itâ€™s not a binary tree, itâ€™s a binary search tree
  • Multilevel search tree used to implement dynamic multilevel indexing
  • Primary disadvantage of implementing multilevel indexes is that the performance degrades as file grows
  • Can be remedied by reorganization but frequent reorganization is not advisable
  • It is dynamic and the adjustment/reorganization when you delete or insert records, it is implicit as in itâ€™s automatically taken care by this structure

B+ Tree of Order p

  • Any node can contain max of p pointers
  • Minimum number of pointers any node can contain is ceil(p/2)
  • That means it contains p-1 keys
  • Note: In a B+ tree record pointer for a record with given key can be found only at leaf node. But if it is in case of B-tree it can happen at intermediate node also. Hence in B+ tree search, success or failure can be declared only after reaching leaf level. Where as in B-tree search can be successful at intermediate level as well. On failure we reach the leaf level.
  • Leaf level pointers will actually point to the data block

B Tree vs B+ Tree

B Tree

  • Here we have data pointers from the nodes also but in B+ trees this is not there
  • Not necessary to reach the leaf node to declare success of the search
  • But for failure you have to reach in B tree also
  • Deletion is very complex compared to B+ tree

B+ Tree

  • Appearance of a key more than once is possible in B+ Tree but not in B tree
  • Easier to maintain and manage

Bitmap Index

  • Simply an array of bits
  • Bitmap index on attribute a of relation r consists of one bitmap for each value that A can take
  • Small and occupy less space
  • Useful when we retrieve records on multiple search keys and also perform aggregate operations
