Introduction

  • Variety of Programs use Batch System(Jobs) or Time Shared Systems(User Programs/Tasks)
  • Terms jobs and processes used interchangeably
  • Process â†’ a program in execution and process execution happens in a sequential fashion
    • text section â†’ Program Code
    • Program counter â†’ Processor Registers
    • Stack â†’ temporary data for function parameters, return addresses, local variables
    • Data Section â†’ global variables
    • Heap â†’ containing memory dynamically allocated during run time
  • Program â†’ passive entity â†’ process is active
    • Becomes process when exe is loaded into memory

States of a Process

  • New â†’ process is being created
  • Running â†’ process is being executed
  • Waiting â†’ process is waiting for some event to occur
  • Ready â†’ process is waiting to be assigned to a processor
  • Terminated â†’ process finished execution

NOTE: No way of going from waiting to running

Process Control Block â†’ PCB

  1. Process state â€“ new, ready, running, waiting, etc.
  1. Program counter â€“ location of instruction to next execute
  1. CPU registers â€“ contents of all process-centric registers
  1. CPU scheduling information- priorities, scheduling queue pointers, scheduling parameters
  1. Memory-management information â€“ memory allocated to the process
  1. Accounting information â€“ CPU used, clock time elapsed since start, time limits, account nos., process nos.
  1. I/O status information â€“ I/O devices allocated to process, list of open files

Process Creation

  • Parent process creates children processes and so on until a tree of processes arrives
  • Process Identifier â†’ PID â†’ integer number
  • Resource Sharing options
    • Parent and children share all resources
    • Children share a subset of parentâ€™s resources
    • Parent and child share no resources
  • Execution Options
    • Parent and children execute concurrently
    • Parent waits until children terminate
  • Address Space
    • Child duplicate of the parent and child has a program loaded in it

fork()

  • Child is a copy of the parent
  • Both child and parent continue execution at the instruction after fork
  • Returns 0 for the child and non zero for the parent

exec()

  • loads a binary file into memory and starts execution
  • destroys previous memory image
  • call to exec does not return
  • returns -1 if not possible or error

wait()

  • parent can issue wait() to move out of the ready queue until the child is done

Process Termination

  • Process executes last statement and then asks the operating system to delete it using the exit() system call.
  • May return status data from child to parent (via wait())
  • Processâ€™ resources are deallocated by operating system
  • Parent may terminate the execution of children processes because:
    • Child has exceeded allocated resources limit
    • Task assigned to child is no longer required
    • Parent is exiting and the operating systems does not allow a child to continue if its parent terminate
  • If a process terminates, then all its children must also be terminated - cascading termination - All children, grandchildren, etc. are terminated.
  • The termination is initiated by the operating system.
  • The parent process may wait for termination of a child process by using the wait() system call. The call returns status information and the pid of the terminated process 
pid_t pid;
int status;
pid = wait(&status); //parent can tell which child has terminated
  • If no parent waiting (did not invoke wait() till then) process is a zombie (if child terminates and parent doesnâ€™t call wait() or calls wait() after some time, in the meantime the child will be in a state called zombie state, where it has terminated but its entry is still present in process table.
    • Here basically child terminated and is stuck in the ps table but parent hasnâ€™t called wait
  • When wait() is called, its entry gets deleted and it doesnâ€™t remain zombie anymore)
  • If parent terminated, process is an orphan (when child is still running but parent terminates without calling wait(), then that child process is called orphan process. Orphan process will eventually be converted to zombie once it is terminated.)
#include <sys/types.h>
#include <unistd.h>
#include <stdio.h>
#include <stdlib.h>

// Zombie means parent does not wait for the process to die
// so the process is never removed from the queue
// in this case if wait is never called then the os will remove the process as
// part of it's clean up
int main()
{
    pid_t pid = fork();

    if (pid == 0)
    {
        printf("In child %d\n", getpid());
    }
    else
    {
        printf("In Parent %d\n", getpid());
        printf("Process will stay in zombie mode\n");
        sleep(10);
        printf("Process will leave zombie mode\n");
        wait();
        sleep(10);
    }
}

Why IPC?

  • Processes within a system may be independent or cooperating
  • Cooperating process can affect or be affected by other processes, including sharing data
  • Reasons for cooperating processes:
    • Information sharing
    • Computation speedup
    • Modularity
    • Convenience
  • Cooperating processes need inter process communication (IPC). Two models of IPC
    • Shared memory:
      • It is faster than message passing
      • Large amount information can be shared
      • One process creates shared memory segment and other processes link/attach to that memory segment to communicate with each other (creation, attaching, de-attaching will require system call but writing/reading in shared memory are normal/donâ€™t require system call)
      • Two process are not allowed to write simultaneously in the shared memory segment, but two/more can read simultaneously
    • Message passing
      • Only small chunk of data can be communicated

Message Passing

  • Mechanism for processes to communicate and to synchronize their actions
  • Processes communicate with each other without resorting to shared variables, no sharing of address space
  • IPC facility provides two operations:
    • send(message)
    • receive(message)
  • The message size is either fixed or variable
  • If processes P and Q wish to communicate, they need to:
    • Establish a communication link between them
    • Exchange messages via send() / receive()

Direct Communication

  • Processes must name each other explicitly:
  • send (P, message) â€“ send a message to process P
  • receive(Q, message) â€“ receive a message from process Q
  • Properties of communication link
    • Links are established automatically
    • Processes only need to know each otherâ€™s identity
    • A link is associated with exactly one pair of communicating processes
    • Between each pair there exists exactly one link

Indirect Communication

  • Messages are directed and received from mailboxes (also referred to as ports):
  • Each mailbox has a unique ID
  • Processes can communicate only if they share a mailbox
  • Properties of communication link
    • Link is established only if processes share a common mailbox
    • A link may be associated with many processes
    • Each pair of processes may share several communication links, each link corresponds to one mailbox

Synchronization

  • Message passing may be either blocking or non blocking
  • Blocking is considered synchronous
    • Blocking send - the sender is blocked until the message is received by the receiving process or mailbox
    • Blocking receive - the receiver is blocked until a message is available
  • Non blocking is considered asynchronous
    • Non blocking send - the sender sends the message and continues
    • Non blocking receive - the receiver receives
      • A valid message
      • Null message

Buffering

  • Temporary queue stores these messages
  • Zero Capacity â†’ No messages are queued, link canâ€™t have any waiting messages, sender is blocked until receiver receives message
  • Bounded Capacity â†’ Queue is of finite length n messages, sender is blocked if the queue becomes full
  • Unbounded Capacity â†’ Sender is never blocked

Pipe

  • Acts as a conduit allowing two processes to communicate

Ordinary Pipes

  • NOTE: Cannot be accessed from outside the process that created it
  • Parent process creates a pipe and uses it to communicate with a child process
  • Allow communication in standard producer - consumer style
  • Producer writes to one end - the write end and the consumer reads at another end - the read end
  • Windows calls these anonymous pipes
  • We count pipe as a regular file â†’ so that we can use everything as a file in c read and write functions
  • These are unidirectional
  • NOTE: We need to close read end if we have to use write end

Named Pipes

  • More powerful and bidirectional
  • No parent child relationship
  • Do not cease to exist if communicating processes terminate
  • FIFO in UNIX systems

Message Queues

  • Asynchronous communications
  • Messages placed onto queue are stored until recipient receives them
    • msgget() â†’ connect
    • msgsnd() â†’ send
    • msgrcv() â†’ receive
    • msgctl() â†’ control and delete

Context Switch

  • System must save old process state and loads the new process state
  • Context of a process in PCB (CPU registers contents, process state, memory management information)
  • Context Switch Time is overhead, system does no useful work while switching
  • Time is dependent on hardware support

Process Scheduling

  • Maximize CPU use quickly switch processes onto CPU for time sharing
  • Process scheduler selects among available processes for next execution
  • Maintains scheduling queues
    • Job Queue â†’ Set of all processes in the system â†’ secondary memory
    • Ready Queue â†’ Set of all processes in main memory, ready and waiting to execute, generally â†’ Linked List
    • Device Queues â†’ Waiting for an IO device (Main memory)
  • Process migrate among the various queues

Schedulers

Short term Scheduler

  • Invoked frequently and must be fast
  • Selects which process should be executed next and allocates CPU
  • Sometimes the only scheduler in a system

Long Term Scheduler

  • Selects which processes from the job queue should be brought into the ready queue
  • Invoked infrequently (secs, mins) â†’ may be slow
  • Controls the degree of monoprogramming â†’ no of processes in main memory
    • IO bound process â†’ spends more time doing IO than computations, many short CPU bursts â†’ amount of time a process executes on the CPU
    • CPU bound process â†’ spends more time doing computations, few very long CPU bursts
  • Strives for good process mix

Medium Term Scheduler

  • Can be added in time sharing systems, if degree of multiprogramming to decrease
  • Remove process form memory, store on disk, bring back in from disk to continue execution from where it left off
  • Required for improving process mix or for freeing of memory
